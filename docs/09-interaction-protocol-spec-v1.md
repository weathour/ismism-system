# ISMISM 交互协议规格（v1）

## 0. 本文讨论什么

这份文档只讨论产品层面的交互协议，不讨论具体 API 路由、前端组件实现、序列化细节或模型推理代码。

它回答的问题是：
- 系统支持哪些高价值交互协议
- 每种交互的输入和输出应该长什么样
- LLM 输出不应只是一段自然语言，而应有哪些结构
- 卡片、地图、会话、用户系统如何在一次交互中协同

一句话结论：

> ISMISM 的交互协议必须是“结构化理论工作流协议”，而不是自由散漫的聊天协议。

也就是说：
- 允许自然语言输入
- 但系统内部必须转成结构化任务
- 输出也必须至少部分结构化

---

## 1. 为什么需要交互协议

如果没有明确协议，系统会退化成：
- 用户随便问一句
- LLM 随便答一段
- 地图、用户系统、案例库、证据卡都变成摆设

而 ISMISM 的系统目标恰恰是：
- 学习
- 判定
- 比较
- 协作
- 推演

这些任务都不是“纯文本回答”能稳定承载的。

所以交互协议的作用是：
- 把自然语言问题转成明确任务
- 让 LLM 有结构化输出目标
- 让地图、证据、候选、卡片都能挂上去

---

## 2. 交互协议的总体原则

### 原则 1：输入可自然语言，内部必须结构化
用户可以用自然语言发问，但系统内部必须识别：
- 任务类型
- 对象类型
- 目标层级
- 期望输出深度

### 原则 2：输出默认带“候选 + 证据 + 下一步”
即使界面展示为自然语言，系统内部也应结构化保存：
- 候选
- 证据
- 边界
- 下一步跳转

### 原则 3：协议必须支持关系型思维
ISMISM 不是静态节点库，而是高度重视：
- 运动
- 矛盾
- 中介
- 短路
- 跃迁
- 推演

因此协议不能只围绕“单节点说明”，还要支持：
- 节点与节点的关系分析
- 路径分析
- 矛盾/运动分析

### 原则 4：协议必须与地图和会话状态对齐
任何高价值交互都要能说明：
- 当前 thread 是什么
- 当前活跃对象是什么
- 当前推荐跳到哪里
- 当前候选和冲突节点是什么

### 原则 5：协议必须兼容用户档位
同样一个问题，在：
- 学徒档
- 研究档
- 学者档

输出结构应保持同构，但详略不同。

---

## 3. 核心任务类型

建议系统内部至少识别八类任务。

## 3.1 Learn（学习）
典型用户意图：
- 这是什么？
- 我该先学什么？
- 帮我进入这个分支

## 3.2 Orient（定位）
典型用户意图：
- 这个点在整个体系里在哪里？
- 它和什么相邻？

## 3.3 Diagnose（判定）
典型用户意图：
- 这个对象更像哪个格位？
- 帮我给一个 provisional judgment

## 3.4 Compare（比较）
典型用户意图：
- 它更像 A 还是 B？
- 这两个点真正差在哪？

## 3.5 Trace（追踪）
典型用户意图：
- 这个节点是怎么运动到另一个节点的？
- 它的跃迁链是什么？

## 3.6 Tension（矛盾/张力分析）
典型用户意图：
- 这里的核心矛盾是什么？
- 哪两个维度在打架？

## 3.7 Project（推演）
典型用户意图：
- 如果接受这个位置，会导向什么主体/组织/实践？

## 3.8 Revise（协作修正）
典型用户意图：
- 我不同意这个候选
- 重新排一下
- 这个边界规则要修

---

## 4. 输入协议：系统需要从自然语言中抽取什么

每次用户发起 heavy action，系统内部应至少抽取以下字段。

### 4.1 `task_type`
枚举值：
- learn
- orient
- diagnose
- compare
- trace
- tension
- project
- revise

### 4.2 `object_type`
枚举值：
- thinker
- doctrine
- text
- corpus_node
- method_node
- case
- subject_position
- organization_form
- practice_path
- relation

### 4.3 `object_ref`
指用户到底在谈什么。
它可以是：
- 一个节点 ID
- 一个案例 ID
- 一段文本
- 一个临时自然语言对象描述

### 4.4 `target_scope`
表示用户要判多大范围：
- overall
- one_dimension_only
- local_branch
- relation_only
- route_only

### 4.5 `response_depth`
表示用户要多深：
- quick
- standard
- deep
- scholar

### 4.6 `relation_focus`
这在 ISMISM 里尤其重要。
应支持：
- none
- contradiction
- mediation
- short_circuit
- transition
- projection
- confusion
- multiple

### 4.7 `user_mode`
来自用户系统或当前手动切换：
- learner
- researcher
- scholar

### 4.8 `thread_context_ref`
系统需要知道当前线程正在围绕什么工作。

---

## 5. 输出协议：所有重型交互的共用骨架

无论是哪一种任务，输出至少应有一个共用结构。

建议统一骨架：

- `task_summary`
- `object_summary`
- `primary_output`
- `candidate_outputs`
- `relation_findings`
- `evidence_pack`
- `uncertainty`
- `recommended_jumps`
- `next_action_suggestion`

说明如下。

### 5.1 `task_summary`
系统理解你要做什么。

### 5.2 `object_summary`
系统理解对象是什么、位于什么层级。

### 5.3 `primary_output`
本轮的主输出，例如：
- 一条导学路径
- 第一候选判断
- 一个比较结论
- 一条运动链
- 一个张力描述

### 5.4 `candidate_outputs`
尤其在 diagnose / compare / revise 中必须有。

至少支持：
- 第一候选
- 第二候选
- 候选分歧点

### 5.5 `relation_findings`
这是 ISMISM 的特色区。

用于记录：
- 核心矛盾
- 中介结构
- 短路关系
- 运动方向
- 跃迁路径
- 投射到实践层的路径

### 5.6 `evidence_pack`
至少应包含：
- 语料证据
- 方法规则证据
- 边界规则证据
- 案例对照证据

### 5.7 `uncertainty`
不能只给结论，不给不确定性。

### 5.8 `recommended_jumps`
把地图系统接进来。

### 5.9 `next_action_suggestion`
例如：
- 去比较某节点
- 先补某前置概念
- 看某条证据
- 发起 scholar 修正

## 6. 各模式的专用协议

## 6.1 导学模式协议

### 输入关注点
- 用户水平
- 当前目标
- 已学到的位置

### 输出重点
- 推荐起点
- 学习路径
- 先修节点
- 易混淆节点
- 本轮不推荐碰的节点
- 一个小测/对比题

导学模式不应上来就做深度判定。

---

## 6.2 定位模式协议

### 输出重点
- 当前节点位置
- 上位/下位节点
- 相关方法节点
- 常见混淆节点
- 相关案例入口
- 当前推荐跳转

定位模式重点不是解释很长，而是把你放进结构里。

---

## 6.3 判定模式协议

### 输出重点
- 对象
- 判定目标
- 第一候选
- 第二候选
- 关键分歧点
- 支持证据
- 排除理由
- 不确定性
- 下一步建议

### 特别要求
判定模式默认必须允许至少两个候选，除非：
- scholar 明确要求只给单一 provisional 主判定

---

## 6.4 比较模式协议

### 输出重点
- A/B 共同点
- A/B 分野
- 决定性维度
- 常见误判
- 最有区分力的证据
- 建议进一步看哪条边界规则

比较模式默认应联动边界卡。

---

## 6.5 追踪模式协议

这是这套系统非常关键的特色模式。

### 输出重点
- 起点节点
- 终点节点（如果有）
- 中间关键环节
- 触发条件
- 运动方向
- 失败/分叉点
- 相关矛盾

典型问题：
- “3 如何走向 4？”
- “某思想从裂口如何进入主体中介，再走向实践单元？”

追踪模式默认应返回一条“路径对象（Route Object）”。

---

## 6.6 张力 / 矛盾分析协议

### 输出重点
- 主要矛盾是什么
- 矛盾的两端是什么
- 哪一维在承担冲突
- 是否通过某个中介项暂时调和
- 是否已短路
- 是否会逼出下一环节

这是关系型分析的核心入口。

---

## 6.7 推演模式协议

### 输出重点
- 理论位置
- 对应主体形态
- 对应组织倾向
- 对应实践单元倾向
- 可能历史后果
- 风险与偏差

注意：
推演模式默认应比判定模式更保守，因为它更 speculative。

---

## 6.8 协作修正模式协议

### 输入新增内容
- scholar 的修正意见
- 新证据
- 对原候选的否决点

### 输出重点
- 原判断
- 新判断
- 改动点
- 证据差异
- 是否形成新的边界规则 / confusion pair / canonical case 草案

这个模式的目标不是“赢辩论”，而是更新系统状态。

---

## 7. 关系型输出对象必须成为一等公民

你提到一点非常关键：
这个系统显然重视每个节点和其他节点之间的运动与矛盾关系。

所以协议里不能只有“节点说明”，必须让关系型对象成为一等输出。

建议至少支持以下关系型对象：

### 7.1 Route Object
描述：
- 从 A 到 B 的运动路径

字段：
- 起点
- 终点
- 中间环节
- 触发条件
- 分叉点
- 失败点

### 7.2 Tension Object
描述：
- 某个节点或局部系统中的核心矛盾

字段：
- 矛盾双方
- 所在维度
- 当前状态（未调和 / 暂时调和 / 已短路 / 已跃迁）

### 7.3 Mediation Object
描述：
- 某个环节通过什么中介来维持自己

字段：
- 被中介对象
- 中介项
- 中介类型
- 稳定性
- 是否可持续

### 7.4 Contradiction Pair
描述：
- 两个看似相近、实则矛盾或对立的点

### 7.5 Projection Chain
描述：
- 某个理论位置如何投射到主体 / 组织 / 实践层

这些对象非常适合被地图、卡片和预生成资产共同复用。

---

## 8. 与地图系统的接口

交互协议必须能直接产出地图可消费内容。

所以每次 heavy action 的输出都建议至少带：
- `highlight_nodes`
- `highlight_edges`
- `recommended_jumps`
- `blocked_jumps`（当前不建议跳的）
- `relation_objects`

这会让前端不是只显示一段话，而是能同步点亮地图。

---

## 9. 与用户系统的接口

交互协议也必须读取用户系统的关键状态。

建议每次生成时默认读：
- 用户类型
- 当前模式偏好
- 当前掌握度摘要
- 当前高频混淆点
- 当前学习目标

但不应默认读：
- 全量历史对话
- 全量草稿
- 无关个人信息

---

## 10. 与会话系统的接口

交互协议必须服从会话与运行纪律。

也就是说：
- 只有 heavy action 才新建 run
- 同线程只允许一个活跃 heavy run
- 地图浏览和节点查看属于 light action
- 输出应能写回 thread summary

交互协议不是独立存在，它是：
- 用户输入
- 会话状态
- 地图状态
- 用户状态
- LLM 推理
之间的桥梁。

---

## 11. 当前最重要的限制

### 限制 1：不要让协议退化成“聊天模板”
必须有结构对象。

### 限制 2：不要只围绕单节点组织协议
必须支持关系、路径、矛盾、运动。

### 限制 3：不要输出只有自然语言、没有可复用对象
否则地图和卡片无法联动。

### 限制 4：不要对不同用户档位输出完全不同结构
可以调详略，但不要破坏同构。

### 限制 5：不要忽视 scholar 修正会反过来改变资产体系
协议必须允许“修正 -> 资产升级”的链路。

---

## 12. 当前推荐的产品结论

一句话总结：

> ISMISM 的交互协议必须从“问答协议”升级成“结构化理论工作流协议”，并且把关系型对象（运动、矛盾、中介、跃迁）正式纳入输出核心。

这就是这套系统区别于普通 LLM 聊天产品的关键地方。

---

## 13. 下一份文档最适合写什么

继续只做宏观文档、不开始施工的话，下一份最值得写的是：

1. `artifact-lifecycle-and-review-spec-v1`
   - 预生成资产如何从 draft -> reviewed -> canonical
2. `workspace-and-permission-spec-v1`
   - 私有层 / 团队层 / 公共层
3. `relation-object-spec-v1`
   - Route / Tension / Mediation / Projection 的统一对象规范

如果只能先写一个，我现在更倾向于：
- `relation-object-spec-v1`

因为你刚刚指出的重点非常对：
这个系统真正的核心不是节点本身，而是节点之间的运动和矛盾关系。
