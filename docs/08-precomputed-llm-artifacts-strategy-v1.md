# ISMISM 预生成 LLM 衍生文档策略（v1）

## 0. 这份文档回答什么问题

这份文档只讨论一个产品层问题：

> 是否需要提前用 LLM 把整个 ISMISM 语料都“过一遍”，生成大量后续可用的文档资产？

这里不讨论具体实现脚本或模型参数，只讨论策略。

一句话结论：

> 需要提前生成一部分高价值衍生资产，但不应该把“全量语料一次性 LLM 化”作为默认路线。

也就是说：
- 不是完全不预生成
- 也不是先把所有内容都让 LLM 过一遍

最合理的是：
- 分层预生成
- 高价值对象优先
- 其余按需 / 惰性生成

---

## 1. 为什么会想做“全量先跑一遍”

这个冲动是很自然的，因为当前语料规模已经不小：
- `split_md`: 362 篇 markdown
- `split_md_clean`: 362 篇 markdown
- `Atlas_DB`: 9 个 jsonl 派生文件

而系统又想做：
- 地图
- 学习
- 判定
- 比较
- 训练
- scholar 协作

所以直觉会是：
“是不是应该先让 LLM 把所有东西都总结、提炼、结构化，后面才好用？”

这个想法有一半是对的：
- 的确需要一些预生成资产

但如果直接走“全量先跑一遍”路线，会出现很大问题。

---

## 2. 为什么不建议直接全量一次性 LLM 化

## 2.1 成本高，而且很多产物很快会废

如果 362 篇全部先做：
- 摘要
- 关键词
- 分类
- 方法标签
- 边界判断
- 候选格位
- 学习提示

短期看很爽，长期看很可能有大量废件。

原因：
- 体系理解会继续变
- 方法骨架还在显化中
- Scholar 未来会纠偏
- 你现在对产品形态还在调整

所以如果一次性全量生成，很多文档会过早固化错误理解。

## 2.2 会把“活系统”误做成“静态摘要库”

ISMISM 最强的不是摘要，而是判断过程。
如果先把全量内容都转成摘要文档，产品会不自觉地退回：
- 搜摘要
- 看总结
- 聊总结

而不是：
- 做候选
- 看边界
- 跑诊断
- 和学者协作修正

## 2.3 高度依赖当前 prompt 口径

LLM 预生成资产的质量很依赖：
- 当前理解的系统定位
- 当前 prompt
- 当前方法规则是否稳定

如果方法主骨架和边界规则还在成形，一次性全量生成会把“暂时理解”误写成“长期资产”。

## 2.4 维护成本会非常高

你一旦预生成了 362 篇派生摘要，后面会立刻有维护问题：
- 规则变了要不要重跑
- 哪些资产过期了
- 哪些是 scholar 修正过的版本
- 哪些还是旧 prompt 时代的遗产

这会形成第二套负担很重的系统。

---

## 3. 那是不是完全不该预生成？

也不是。

完全不预生成也会有问题：
- 每次都临时让 LLM 从原文现判，成本高
- 学习模式和地图模式缺乏稳定锚点
- 很难形成 canonical case / confusion pair / 边界节点
- 用户系统和地图系统没有足够的稳定对象可挂接

所以正确答案不是“要不要预生成”，而是：

> 预生成什么、预生成到哪一层、哪些必须稳定、哪些必须惰性。

---

## 4. 推荐的总体策略：三层衍生资产策略

建议把衍生资产分成三层。

## 4.1 Layer A：必须预生成的稳定资产

这类资产应该优先做，而且尽量少依赖开放式主观解释。

适合预生成的内容：
- 结构性索引
- 节点目录标准化信息
- 节点间基础关系骨架
- 章节级 summary seed（简短）
- 方法主轴锚点
- canonical case 基础卡
- confusion pair 基础卡
- 证据锚点索引

这类资产的特点：
- 高复用
- 相对稳定
- 为地图和检索服务
- 可以给后续 LLM 调用当上下文锚点

## 4.2 Layer B：建议半预生成的可修正资产

这类资产有价值，但不能一次性做死。

适合内容：
- 节点解释卡
- 边界说明卡
- 常见误判卡
- 初版候选格位说明
- 实践导向草稿
- 学习路径节点说明

这类资产的特点：
- 很有用
- 但容易随着体系理解变化而变
- 适合做“草稿层资产”

## 4.3 Layer C：不建议全量预生成、适合按需生成的资产

适合按需生成的内容：
- 某个用户当前问题的专门诊断报告
- 两个节点的临时比较报告
- 某次 scholar 争议的详细分析
- 面向某个用户学习阶段的个性化解释
- 某次实践推演输出

这类资产的特点：
- 强上下文依赖
- 强个体依赖
- 强任务依赖
- 不适合批量预生成

---

## 5. 具体来说，哪些东西值得提前让 LLM 过一遍

我建议优先考虑下面这些。

## 5.1 节点摘要种子（Summary Seed）

不是长摘要，而是短种子。

建议每个高价值节点预生成：
- 一句话定义
- 两三句功能说明
- 它在体系里的位置
- 一两个容易混淆对象

这会非常适合挂在地图节点卡上。

## 5.2 方法节点卡

优先为这些节点做高质量预生成：
- 场域论
- 本体论
- 认识论
- 目的论
- 1 / 2 / 3 / 4
- 实践单元
- 边界规则主轴

原因：
这些是高频入口，回报率很高。

## 5.3 canonical cases

这是最值得提前做的资产之一。

每个 canonical case 可以预生成：
- 案例对象
- 为什么是 canonical
- 对应位置
- 关键证据
- 常见误判方向

因为这类资产一旦稳定，会被重复使用。

## 5.4 confusion pairs

这是地图和训练系统最值钱的衍生资产之一。

每个 confusion pair 可以预生成：
- 为什么常混
- 真正分野在哪
- 快速判别标准
- 应优先查看的证据类型

## 5.5 证据锚点索引

不是让 LLM 大段解释，而是让 LLM 辅助从海量文本里找：
- 适合作为该节点核心证据的片段
- 适合作为“反例 / 排除理由”的片段

这会显著提升后续诊断效率。

---

## 6. 哪些东西不该提前全量生成

## 6.1 全量长篇详细摘要

不建议 362 篇全部生成长摘要。
理由：
- 维护成本极高
- 容易变成静态摘要库
- 对交互式系统帮助没有想象中大

## 6.2 全量格位终判

不建议让系统先把所有节点都“终判完”。
理由：
- 体系理解还会变
- 学者未来会修正
- 很多节点更适合保留候选性

## 6.3 全量实践推演

不建议提前给每个节点都做“主体/组织/实践后果”推演。
理由：
- 解释空间太大
- 很容易过拟合当前口径
- 更适合在具体任务里按需展开

## 6.4 面向具体用户的个性化文档

不应提前生成，因为：
- 用户是谁还未知
- 上下文不稳定

---

## 7. 推荐的预生成节奏

我建议采用“主干先、案例先、其余惰性”的节奏。

### 第一步：主干节点与方法节点
先做：
- 1/2/3/4
- 四维框架
- 实践单元
- 边界规则主轴

### 第二步：高价值分支节点
再做：
- 3-4 符号学
- 4-1 政治-经济-意识形态批判
- 4-3 建设理想社会
- 4-4 不可能的乌托邦
- 其他明显高杠杆节点

### 第三步：案例层
再做：
- canonical case
- confusion pair
- hard case

### 第四步：按需扩展
只有当某些区域真的高频被访问、被诊断、被争议时，再追加更多预生成资产。

---

## 8. 如何理解“让 LLM 提前过一遍内容”最合理

如果你现在让我给一个最合理的定义，我会说：

不是“让 LLM 先把所有内容都总结一遍”，
而是：

> 让 LLM 先为系统生产一批高复用的中间知识对象。

关键不在“读完了多少”，而在“生成了哪些后续反复使用的对象”。

这些对象包括：
- 方法卡
- 节点卡
- 案例卡
- 边界卡
- 证据锚点
- 混淆对照卡

这才是对后续产品真正有用的“预生成文档”。

---

## 9. 文档资产的生命周期建议

所有预生成的 LLM 衍生文档都不应被视为永久真理。

建议至少分三种状态：

### A. draft
- 初次由 LLM 生成
- 尚未经过人工或 scholar 审视

### B. reviewed
- 已经过人工或高质量规则校正
- 可被系统反复使用

### C. canonical
- 已被明确认可为长期稳定资产
- 可进入地图、训练、诊断主流程

这个生命周期非常重要。
否则预生成资产会越来越脏。

---

## 10. 与 Atlas 的关系

这里也要顺手说清楚。

如果要预生成新文档，不应简单地把 Atlas 当成最终来源。
更合理的关系是：
- Atlas 提供候选
- LLM 提供可读化与结构化
- Scholar / 系统规则做最终筛选

也就是说：
Atlas 仍然只是辅助层，不能直接替代新的衍生资产体系。

---

## 11. 与地图、用户系统、会话系统的关系

### 与地图系统
预生成资产最适合服务地图：
- 节点卡
- 边界卡
- confusion pair
- 证据锚点

### 与用户系统
预生成资产为用户系统提供：
- 可挂掌握度的节点
- 可训练的案例
- 可复习的知识对象

### 与会话系统
预生成资产能减少每次 run 的上下文负担，降低无意义 LLM 重复劳动。

---

## 12. 当前推荐的产品结论

如果只说一句话：

> 需要提前用 LLM 处理一部分内容，但目标不是“全量摘要化”，而是“提前生产一批高复用、可持续、可修正的中间知识对象”。

换句话说：
- 要预生成
- 但不要全量一次性预生成
- 要分层
- 要有生命周期
- 要优先服务地图、训练和诊断

---

## 13. 现阶段最值得提前生成的第一批资产

如果让我给一个现实可执行的优先级列表，我会建议第一批只做：

1. 方法节点卡
2. 主干节点卡
3. 高价值分支节点卡
4. canonical case 卡
5. confusion pair 卡
6. 证据锚点索引

不要一开始就做：
- 362 篇全量长摘要
- 全量终判
- 全量实践推演

---

## 14. 下一份文档最适合写什么

在继续只做宏观文档、不开始施工的前提下，下一份最适合写：

1. `interaction-protocol-spec-v1`
   - 各模式输入输出契约
2. `workspace-and-permission-spec-v1`
   - 私有层 / 团队层 / 公共层
3. `artifact-lifecycle-and-review-spec-v1`
   - draft / reviewed / canonical 的升级规则

如果只能先写一个，我建议：
- `artifact-lifecycle-and-review-spec-v1`

因为一旦开始做预生成资产，就必须提前定义它们何时算草稿、何时可复用、何时能进主系统。
