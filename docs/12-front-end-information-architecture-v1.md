# ISMISM 前端信息架构规格（v1）

## 0. 这份文档讨论什么

这份文档只讨论 ISMISM 的前端信息架构，不讨论具体前端技术栈、组件库选型、接口协议命名、状态管理库、CSS 方案或数据库表设计。

它回答的问题是：
- 这个系统前端到底应该由哪些页面 / 工作区构成
- 哪些区域应该始终可见，哪些应该按上下文展开
- 地图、对话、证据、资产、运行状态之间如何同屏共存
- 第一版前端不应该长成一个什么都有、但什么都不稳的“万能聊天壳”

一句话结论：

> ISMISM 的前端不应被设计成“一个大聊天框 + 一个图谱按钮”，而应被设计成一个围绕地图、线程、证据与关系型资产展开的理论工作台。

---

## 1. 前端信息架构的总原则

### 原则 1：前端首先是工作台，不是展示站
ISMISM 的主要使用场景不是“来看看这个系统是什么”，而是：
- 进入一个问题
- 沿着结构导航
- 发起一次导学 / 判定 / 比较 / 训练任务
- 查看证据、关系与资产状态
- 在 scholar / research 工作流中修正资产

所以前端的主骨架应该服务于“持续工作”，而不是服务于“漂亮首页”。

### 原则 2：地图、线程、证据必须形成稳定三角
前端不能把地图做成独立孤岛，也不能把线程做成与结构脱钩的纯对话流。

系统的核心工作状态应该始终能在三者之间来回切换：
- 地图：我现在在系统结构里的什么位置
- 线程：我当前在问什么、正在推进什么
- 证据：这个判断依赖什么文本、案例、关系资产

### 原则 3：重推理动作必须显式，轻浏览动作必须常开
根据 `docs/07-session-and-run-state-spec-v1.md`：
- 一个线程同一时间只允许一个 active heavy run
- 但浏览地图、打开卡片、查看证据、切换侧栏这类轻动作应始终可用

因此前端必须把“正在生成”显示为线程级约束，而不是全站死锁。

### 原则 4：模式不同，但工作台骨架应尽量统一
ISMISM 至少有这些模式：
- 导学 / orient
- diagnose
- compare
- tension / trace / project
- training
- scholar / review

它们的任务不同，但不应每个都设计成完全不同的网站。更好的方式是：
- 共享一套工作台骨架
- 中心区域承载不同模式的主面板
- 左右侧栏承载结构、上下文、证据与资产
- 底部 / 顶部承载运行状态与线程动作

### 原则 5：关系型对象的可见性必须高于普通摘要对象
既然本系统强调：
- 运动
- 矛盾
- 中介
- 跃迁
- 短路

那么前端就不能只优待节点摘要卡，而要让下列对象也能被稳定打开、比较与引用：
- Route Card
- Tension Card
- Mediation Card
- Boundary Card
- Short-Circuit Card
- Projection Chain

### 原则 6：前端信息架构必须天然支持生命周期与审核
根据 `docs/11-artifact-lifecycle-and-review-spec-v1.md`，资产存在：
- draft
- reviewed
- canonical
- deprecated
- superseded
- rejected
- disputed-reviewed（建议扩展态）

因此前端不能只展示“内容”，还必须展示：
- 资产状态
- 来源
- 证据锚点
- 审核历史
- 当前是否可进入主流程

## 2. 顶层信息架构

建议把前端分成三层入口，而不是一个单页把所有东西塞进去。

## 2.1 公共入口层

用途：
- 向第一次进入的人解释 ISMISM 是什么
- 让用户选择进入学习 / 诊断 / 研究 / scholar 工作模式
- 提供少量可公开浏览的 canonical 内容

这一层不是主工作面，但仍然需要。

建议包含：
- 系统简介
- 进入工作台按钮
- 几个 canonical 示例入口
- 方法骨架概览
- 权限 / 身份说明

注意：
公共入口层不应承担完整工作流。

## 2.2 登录后工作台层

这是第一核心层，也是绝大多数工作发生的地方。

建议包含：
- 工作台首页
- 地图页
- 线程工作区
- 训练页
- 比较 / 诊断 / 导学等任务页
- 个人工作区与最近历史

这里应采用统一的 shell。

## 2.3 scholar / review 工作层

这是高权限层，不一定与普通用户隔离成完全不同前端，但必须在信息架构上独立。

用途：
- 审核资产
- 比较 draft / reviewed / canonical 差异
- 检查证据回链
- 处理 disputed-reviewed 资产
- 决定升级、废弃、替代

也就是说：
普通工作台解决“如何使用系统”，review 工作层解决“如何维护系统可信度”。

---

## 3. 全局工作台骨架

建议登录后统一采用“五区骨架”。

## 3.1 顶栏（Top Bar）

始终可见。

承载信息：
- 当前系统 / workspace 名称
- 当前模式入口（工作台 / 地图 / 训练 / scholar 等）
- 全局搜索入口
- 当前用户身份与权限
- 当前线程名
- 当前 run 状态摘要
- 快速新建任务按钮

顶栏的任务不是显示很多信息，而是始终让用户知道：
- 我在哪
- 我是谁
- 我当前正在跑什么
- 我能切去哪里

## 3.2 主导航栏（Primary Rail）

建议固定在左侧，始终可见。

建议一级入口：
- 首页 / Workbench
- 地图
- 导学
- 诊断
- 比较
- 训练
- Scholar / Review
- Assets（可选，高权限或高级用户开放）

说明：
这里是“模式切换”，不是“文件树”。

## 3.3 上下文侧栏（Context Rail）

建议位于主导航栏右侧，可折叠。

根据模式切换内容，例如：
- 地图模式：显示地图层级、筛选器、路径列表、已打开对象
- 导学模式：显示当前问题上下文、前置节点、推荐路径
- 诊断模式：显示候选对象、待比较对象、排除项
- Scholar 模式：显示审核队列、状态筛选、待处理资产

也就是说：
一级导航负责“你在做什么”；上下文侧栏负责“你在这个模式下具体看什么”。

## 3.4 主工作区（Main Canvas）

这是核心区域。

它负责承载：
- 地图主画布
- 线程主结果
- 训练题面与作答面板
- 诊断结果视图
- 比较视图
- 审核视图

设计原则：
主工作区一次只应有一个“主任务焦点”。

可以同时保留很多上下文，但中心焦点只能有一个，否则用户会失去推进感。

## 3.5 右侧洞察面板（Insight / Evidence Panel）

建议默认可见，可折叠。

它应承担“所有重要但不应该抢主焦点的信息”，例如：
- 当前对象的摘要卡
- 当前 Route / Tension / Mediation 卡
- 证据锚点列表
- 生命周期状态
- 引用文本
- 上一次 run 的结构化输出对象
- 推荐下一步动作

这个右侧面板非常关键。

因为 ISMISM 不是普通内容阅读器，用户经常需要一边看主流程，一边看关系卡和证据卡。

## 3.6 底部运行托盘（Run Dock）

建议存在，默认收起或半展开。

用途：
- 展示当前线程 run 状态
- 展示队列中是否已有 active heavy run
- 提供 cancel / retry / duplicate / open trace 等动作
- 显示“当前线程已锁定，禁止重复重请求”这类状态

这能把系统纪律显式化，而不是让用户只在按钮失效时猜发生了什么。

## 4. 全局可复用对象与打开方式

不是所有东西都应该做成独立页面。

建议把系统对象分成三类打开方式。

## 4.1 必须可成为独立页面 / 主视图的对象

包括：
- 地图
- 一个线程工作区
- 一次训练任务
- scholar 审核队列
- 资产详情页（高复杂度时）

这些对象复杂度高，适合有完整上下文。

## 4.2 更适合做抽屉 / 侧板的对象

包括：
- 节点卡
- Route Card
- Tension Card
- Mediation Card
- Boundary Card
- 证据锚点集
- 用户掌握状态面板
- 当前线程上下文

原因：
这些对象经常需要被“边看边比”，而不是频繁跳转走。

## 4.3 适合做弹层 / 小面板的对象

包括：
- run 状态提示
- 生命周期状态说明
- 快速跳转建议
- 轻量引用预览
- 证据 hover 预览

原则：
越是高频、短时、辅助性的对象，越不应强迫整页跳转。

## 5. 核心页面族

## 5.1 首页 / Workbench Home

这是登录后的默认落点。

它不是 marketing 首页，而是“我现在该往哪继续”。

首页应回答五个问题：
- 我上次停在哪个线程 / 训练 / 地图位置
- 系统推荐我从哪里继续
- 当前有哪些最近打开的对象
- 我最近的判定 / 比较 / 训练记录是什么
- 当前是否有需要审核或继续处理的资产

建议模块：
- 继续上次工作
- 最近线程
- 最近地图位置
- 推荐训练 / 推荐导学
- 待处理 run / 待审核资产（按权限）
- 系统推荐入口（例如“继续某条路线”）

首页不是为了让用户“浏览内容”，而是为了让用户“重新进入工作状态”。

## 5.2 地图页

地图页是系统的核心认知导航页。

它至少应支持三层：
- 理论真相层
- 用户学习状态层
- 当前任务层

地图页主区建议包含：
- 中央地图画布
- 层切换器
- 筛选器（节点类型 / 边类型 / 生命周期状态 / 难度）
- 当前路径或跳转说明

右侧建议显示：
- 当前选中节点 / 关系卡
- 相关证据
- 相关线程
- 推荐动作（导学 / 比较 / 诊断 / 训练）

地图页最重要的不是“看起来像图”，而是：
- 能展开
- 能收拢
- 能跳转
- 能解释为什么跳
- 能连接到线程与资产

## 5.3 导学页 / Orient Workspace

这不是单独课程视频页，而是围绕“我该怎么进入一个问题”而设计的线程工作区。

适合任务：
- 从一个节点进入
- 从一组困惑进入
- 从一个概念误区进入
- 从一个实践问题进入

页面应包含：
- 当前问题
- 推荐学习路径
- 需要先看的节点 / 关系
- 当前回答流
- 证据与关系卡
- 下一步建议

重点：
导学页应该让用户感觉自己在“被带着进入结构”，而不是在“听一段长解释”。

## 5.4 诊断页 / Diagnose Workspace

适合任务：
- 判断一个对象的理论位置
- 查看候选格位
- 比较支持证据与排除理由
- 输出 provisional 或 reviewed 级判断

建议主区包含：
- 诊断对象输入区
- 当前候选位置
- 支持证据
- 排除链
- 不确定性说明
- 相关 Route / Tension / Boundary 资产

建议右侧始终可见：
- 证据锚点
- 相似案例
- 可对照对象
- 生命周期状态

诊断页必须清楚区分：
- 已确认
- 候选
- 被排除
- 待补证据

不能输出一个看似整洁但来源不透明的“神秘最终结论”。

## 5.5 比较页 / Compare Workspace

适合任务：
- 比较两个节点 / 思想家 / 路径 / 实践姿态
- 解释相似与差异
- 找出边界、矛盾、中介与短路点

建议主区采用双列或多段比较视图：
- 对象 A
- 对象 B
- 中间结构层（共性 / 差异 / 边界 / 转化关系）

建议右侧显示：
- 对应 Route / Tension / Boundary 卡
- 证据对应关系
- 推荐进一步比较对象

比较页不应只是“两段摘要并排”，而应把关系对象放在中间。

## 5.6 训练页 / Training Workspace

适合任务：
- 做题
- 自测
- 区分相似对象
- 进行纠偏
- 查看 canonical / hard / disputed case

建议信息分区：
- 题目 / 案例区
- 作答区
- 系统反馈区
- 证据区
- 纠偏解释区

训练页最重要的是让用户感受到：
- 错在哪里
- 为什么错
- 应该回到哪条路径补结构

因此训练页必须和地图、导学页互相可跳。

## 5.7 Scholar / Review 页

这是维护系统可信度的关键页。

适合任务：
- 审核 draft 资产
- 检查 reviewed 候选
- 比较版本差异
- 决定 canonical / rejected / superseded
- 处理争议资产

建议包含：
- 审核队列
- 生命周期筛选器
- 证据回链检查面板
- 当前资产详情
- 历史版本与 review note
- 升级 / 降级 / 废弃动作

这里最好不是简单列表页，而是“审核工作台”。

## 5.8 通用线程页 / Thread Workspace

虽然导学、诊断、比较等可以有各自入口，但底层建议统一为线程工作区模型。

线程页应始终显示：
- 线程标题
- 线程模式
- 当前 run 状态
- 线程上下文对象
- 结构化输出历史
- 可再次发起的动作

这样可以避免系统最终变成多个互不兼容的孤立页面。

## 6. 模式切换与线程模型

前端要清楚区分三件事：
- 模式（mode）
- 页面（page）
- 线程（thread）

### 模式
表示当前任务类型，例如：
- orient
- diagnose
- compare
- training
- scholar-review

### 页面
表示当前前端落点，例如：
- 工作台首页
- 地图页
- 某个线程页
- scholar 审核页

### 线程
表示某次连续工作上下文。

也就是说：
- 你可以在 diagnose 模式下开一个线程
- 也可以在 compare 模式下开另一个线程
- 两者都能共享同一个工作台骨架

这套区分非常重要，它会显著减少施工期前端混乱。

---

## 7. 哪些信息应该始终可见

建议下列信息尽量始终可见，或最多一键可见。

## 7.1 当前线程身份
- 当前线程名称
- 当前模式
- 当前上下文对象数
- 当前是否有 active heavy run

## 7.2 当前 run 状态
- idle / queued / running / succeeded / failed / cancelled
- 最近一次更新时间
- cancel / retry 是否可用

## 7.3 当前对象的最小上下文
- 当前选中节点或关系对象名
- 其生命周期状态
- 它与当前线程的关系

## 7.4 可回退路径
- breadcrumb
- 最近打开对象
- 最近地图路径

如果这些都隐藏得太深，用户会非常容易在复杂结构里迷路。

---

## 8. 哪些信息不应该默认常驻主画面

以下信息不应默认把主画面压满：
- 全量原文长摘录
- 全部历史 run 日志
- 全站所有筛选器
- 全部二级关系
- 所有候选资产列表

更好的做法是：
- 默认显示当前最相关的少量信息
- 其余通过抽屉、过滤器、展开区获取

原则：
系统复杂，不等于每个时刻都要显示复杂。

---

## 9. 页面之间的核心跳转

前端的价值不只在页面本身，也在跳转设计。

建议预设以下高频跳转：

### 地图 -> 导学
当用户在地图上选中一个高难节点、关系或困惑区域时，可直接发起：
- 从这里开始导学
- 解释这条路线
- 为什么这里会混淆

### 地图 -> 诊断
当用户想判断某对象在结构中的位置时，可直接带着当前地图上下文进入诊断。

### 诊断 -> 比较
当诊断结果出现两个主要候选位时，可一键进入比较工作区。

### 比较 -> 地图
比较后应能直接返回地图并高亮两者所在区段与中间关系。

### 训练 -> 导学 / 地图
答错后应能直接跳到：
- 对应误区
- 应补的前置节点
- 相应 Route / Boundary 卡

### Scholar -> 资产详情 -> 地图 / 线程
审核不应脱离实际使用场景，因此 scholar 页必须能回看该资产在地图和线程中的呈现位置。

## 10. 第一版最推荐的前端页面集合

虽然理想上页面很多，但 v1 不需要每个模式都拆成完全独立站点。

我建议第一版优先稳定以下 6 类前端页面：

1. 登录后工作台首页
2. 地图页
3. 通用线程工作区页（承载 orient / diagnose / compare / trace / tension 等）
4. 训练页
5. Scholar / Review 页
6. 资产详情侧板 / 详情页

也就是说：
第一版完全可以让：
- diagnose
- compare
- orient
- trace
- tension
都共享一个线程工作区骨架，只是在主面板模板上不同。

这比一开始就拆出很多独立页面更稳。

## 11. 明确不推荐的前端误区

### 误区 1：把系统做成一个聊天首页
后果：
- 结构感消失
- 地图退化为附件
- 关系资产无法稳定工作

### 误区 2：把地图做成只可看不可用的大图
后果：
- 用户无法从地图发起任务
- 地图与线程断裂

### 误区 3：把 scholar 审核混在普通用户界面里
后果：
- 生命周期状态失去边界
- 普通用户会被噪音淹没

### 误区 4：把证据面板藏得太深
后果：
- 诊断与比较失去可回溯性
- 结果看起来像黑箱判断

### 误区 5：让 run 锁只是“按钮变灰”
后果：
- 用户不知道发生了什么
- 线程纪律被误解为系统卡死

## 13. 当前推荐的产品结论

一句话总结：

> ISMISM 的前端应该被做成一个以地图、线程、证据、关系资产和运行纪律为核心的理论工作台；第一版要先把统一工作台骨架做稳，而不是急着做成一个华丽但松散的多页面内容站。
