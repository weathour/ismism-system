# ISMISM 会话与运行状态规格（v1）

## 0. 本文讨论什么

这份文档只讨论产品层面的会话（session / thread）与运行（run）状态，不讨论具体前端框架、消息协议格式或后端实现细节。

它回答的问题是：
- 系统中的“线程”是什么
- 一次 LLM 生成任务是什么
- 为什么必须禁止重复发请求
- 哪些动作应该被锁住，哪些不该被锁住
- cancel / retry / idempotency 的产品语义是什么

一句话结论：

> ISMISM 不是普通网页聊天框，而是一个需要严格会话纪律的理论工作台；同一线程里，在当前 LLM 回答完成前，禁止发起新的 LLM 推理请求。

---

## 1. 为什么必须单独设计会话与运行状态

如果不单独设计这套状态，系统会出现四类问题：

1. 用户双击 / 连击导致重复推理
2. 一个尚未完成的判定被第二个问题打断，上下文语义漂移
3. 地图浏览、轻量查看与重型 LLM 推理混杂，界面显得又卡又乱
4. 后续学者协作时，很难判断哪一轮输出对应哪个对象、哪个候选和哪次纠偏

而 ISMISM 的任务天然是：
- 多轮判断
- 候选比较
- 证据收束
- 学者纠偏

所以“线程与运行状态”不是工程细节，而是产品纪律。

---

## 2. 核心对象定义

## 2.1 Thread（线程 / 会话）

线程是一次连续工作的工作面。

一个线程通常围绕：
- 一个学习目标
- 一个判定对象
- 一个比较问题
- 一个案例训练任务
- 一段 scholar 协作讨论

线程的意义是：
- 把上下文收束在一个工作面里
- 明确当前正在推进什么问题

## 2.2 Run（运行 / 生成任务）

run 指一次已经提交给系统并可能触发 LLM 推理的任务。

一个 run 可能是：
- 生成一份导学路径
- 对某个对象做 provisional diagnosis
- 对两个候选格位做比较
- 根据学者纠偏重排候选
- 生成一份结构化诊断报告

重要区别：
- thread 是容器
- run 是 thread 内的一次推理任务

## 2.3 Light Action（轻动作）

轻动作指不需要 LLM 推理的操作，例如：
- 展开地图节点
- 切换视图
- 浏览证据卡
- 打开某个历史报告
- 切换标签页

## 2.4 Heavy Action（重动作）

重动作指会触发 LLM 推理或需要完整上下文收束的动作，例如：
- 发起诊断
- 发起比较
- 请求解释一段边界差异
- 发起 scholar 模式分析
- 根据新反馈重跑候选判断

---

## 3. 基本纪律：一个线程同一时刻只允许一个活跃重动作

这是整个系统的硬规则。

### 规则本体
在同一 `thread` 中，如果当前已有一个活跃 run 处于：
- preparing
- generating
- streaming
- finalizing

则不允许发起新的 heavy action。

### 产品理由
原因不是“实现麻烦”，而是认知一致性：
- 这个系统输出的是判断过程
- 判断过程强依赖当前对象与当前候选
- 在一个未完成的判定上再开一个新判定，会让语义极不稳定

所以最稳妥的策略是：
- 一个线程，一次只做一件 LLM 重任务

---

## 4. 线程状态机

建议线程至少有以下状态。

## 4.1 `idle`
含义：
- 空闲
- 当前线程没有活跃 run
- 可发起新的 heavy action

## 4.2 `preparing`
含义：
- 系统正在整理上下文
- 拉取相关节点、证据、方法规则、用户状态
- 尚未正式进入模型生成

用户感受：
- 已经提交
- 正在准备回答

## 4.3 `generating`
含义：
- 请求已正式发给 LLM
- 等待首 token / 首段输出

## 4.4 `streaming`
含义：
- 正在流式输出
- 当前回答尚未结束

## 4.5 `finalizing`
含义：
- 模型文本基本生成完毕
- 系统正在整理结构化输出、写入会话摘要、保存 run 结果

## 4.6 `done`
含义：
- 当前 run 完成
- 线程可进入下一轮 heavy action

## 4.7 `error`
含义：
- 当前 run 失败
- 可选择 retry 或放弃

## 4.8 `cancelled`
含义：
- 用户主动停止
- 系统主动中止
- 本轮不再继续流式输出

---

## 5. Run 状态机

run 的状态可以与 thread 状态相近，但应作为独立对象存在。

建议至少有：
- queued（可选）
- preparing
- generating
- streaming
- finalizing
- done
- error
- cancelled

注意：
当前产品版本不建议允许同线程内 run 排队。
所以 `queued` 不是必需，除非以后支持后台计划队列。

---

## 6. 前端交互规则

## 6.1 当线程处于活跃重动作状态时，前端必须：

- 禁用发送按钮
- 禁用 Enter 直接提交
- 禁用所有会触发 heavy action 的入口按钮
- 保留输入框编辑能力（可选）
- 显示当前状态（正在准备 / 正在生成 / 正在整理）
- 提供“停止生成”按钮

## 6.2 当线程处于活跃状态时，前端仍应允许：

- 浏览地图
- 展开节点
- 切换视图
- 查看证据卡
- 查看历史 run
- 浏览已生成内容

也就是说：
- 浏览不断
- 推理锁住

## 6.3 前端状态提示要尽量明确

不要只显示一个模糊的“加载中”。
更好的提示是：
- 正在整理证据
- 正在生成候选判断
- 正在输出中
- 正在保存本轮结果

这样用户能知道系统当前正在做什么。

---

## 7. 后端保护规则

不能只靠前端禁按钮，后端必须有硬保护。

## 7.1 最低必要字段

每个线程至少应维护：
- `thread_id`
- `active_run_id`
- `thread_status`
- `started_at`
- `updated_at`

每个 run 至少应维护：
- `run_id`
- `thread_id`
- `run_type`
- `status`
- `started_at`
- `ended_at`
- `idempotency_key`

## 7.2 重复请求拦截规则

如果后端收到某个 `thread_id` 的新 heavy action，而该线程当前状态仍是活跃中，则：
- 不启动新的 run
- 返回明确冲突错误
- 附带 `active_run_id`
- 告诉前端当前已有进行中的任务

产品层语义建议等价于：
- “当前线程正在回答，请等待或停止当前回答后再试。”

## 7.3 不要默认线程内排队

当前版本建议：
- 同线程不排队

原因：
- 第二个问题往往语义上依赖第一个答案
- 若直接排队，用户会误以为系统理解了“第二个问题是接着刚才那个未完成回答来的”
- 但实际上第一个回答还没完成，第二个问题的真正意图未稳定

所以：
- 要么等
- 要么取消当前 run
- 要么新开线程

---

## 8. Idempotency（幂等）规则

这对产品体验非常重要。

## 8.1 为什么需要幂等键

现实里会发生：
- 用户双击
- 网络重传
- 前端刷新后重发
- 手机端/桌面端重复提交

如果没有幂等保护，就会重复触发 LLM。

## 8.2 基本规则

每个 heavy action 提交时都应附带：
- `idempotency_key`

后端语义：
- 同一个 key 的重复提交，不产生新的 run
- 只返回当前已有 run 的状态、结果或流句柄

## 8.3 幂等窗口

产品层面建议：
- 幂等键至少在本轮 run 生命周期内有效
- 若 run 已结束，可视策略决定短期保留还是释放

---

## 9. Cancel（取消）规则

取消必须被视为产品一级能力，而不是调试功能。

## 9.1 用户应能在活跃状态中主动取消

可取消阶段：
- preparing
- generating
- streaming
- finalizing（可选，视实现）

## 9.2 取消后的语义

取消不等于失败。

取消后应：
- 将 run 标记为 `cancelled`
- 将 thread 恢复到可继续使用状态
- 明确告诉用户：当前轮已中止，可重新发起请求

## 9.3 取消后的内容处理

如果已生成部分内容，产品上建议：
- 可保留部分输出
- 但要清晰标记“本轮未完成”
- 不应把未完成输出冒充最终诊断报告

---

## 10. Retry（重试）规则

重试必须区分两类。

## 10.1 系统级重试

适用：
- 网络波动
- 模型网关偶发错误
- 流中断

特点：
- 用户无需重新组织问题
- 系统可在短窗口内自动重试

## 10.2 用户级重试

适用：
- 本轮错误结束
- 用户想重新跑一轮
- 学者纠偏后想重跑

特点：
- 会形成新 run
- 不应覆盖旧 run
- 应与旧 run 建立关联

---

## 11. 新线程 vs 同线程继续

这是一个很关键的产品问题。

建议规则：

## 11.1 适合同线程继续

当用户是在：
- 追问当前对象
- 要求展开某个候选
- 要求看证据
- 要求改成比较模式
- 在 scholar 模式里做局部修正

这时应继续当前线程。

## 11.2 适合新开线程

当用户是在：
- 切换到完全不同对象
- 从学习模式切到另一个独立任务
- 开始新的案例训练
- 开始另一条 scholar 争议线

这时建议提示新开线程。

原因：
线程本身就是工作面，不应无限膨胀。

---

## 12. 地图操作与会话状态的边界

这一点和地图系统规格必须对齐。

## 12.1 不触发 LLM 的地图动作

以下动作应视为 light action：
- 展开 / 折叠节点
- 查看节点卡
- 查看掌握状态
- 切换地图视图
- 查看推荐跳转列表
- 查看历史 run 摘要

这些动作不受 active run 锁限制。

## 12.2 触发 LLM 的地图动作

以下动作应视为 heavy action：
- 基于当前节点发起诊断
- 基于两个节点发起比较
- 基于当前节点请求“为什么不是 B”
- 基于当前节点发起 scholar 协作分析
- 基于当前节点请求实践推演

这些动作必须受线程锁限制。

---

## 13. 会话历史应该保存到什么粒度

建议不要把每轮对话原样无限膨胀保存。

更推荐保存三层：

### 13.1 消息层
- 原始用户输入
- 原始系统输出

### 13.2 结构层
- 本轮对象
- 本轮模式
- 本轮候选
- 本轮分歧点
- 本轮证据索引

### 13.3 摘要层
- 本轮结论摘要
- 本轮未解决问题
- 与前一轮的关系

这样既可回放，也不会无限重复上下文。

---

## 14. 失败场景设计

系统至少要能明确处理以下失败场景：

1. 用户双击提交
2. 网络抖动导致重复请求
3. LLM 网关超时
4. 用户中途取消
5. 前端刷新后恢复现场
6. 生成成功但 finalizing 失败
7. 线程长期悬挂在活跃状态

这些场景在产品层都必须有语义，不应交给用户自己猜。

---

## 15. 最重要的产品限制

### 限制 1
不能让“继续输入”直接变成新的重型请求。

### 限制 2
不能把线程当作无限长的杂乱聊天记录。

### 限制 3
不能把地图浏览和推理锁绑死。
浏览应持续可用，推理才锁。

### 限制 4
不能让取消后的未完成文本伪装成最终结果。

### 限制 5
不能把重复请求拦截只放在前端。
后端必须硬拦截。

---

## 16. 当前推荐的产品结论

如果只说一句话：

> ISMISM 的会话系统应被设计成“可浏览但不可并发重推理”的单线程工作台模型。

也就是：
- 一个线程 = 一个工作面
- 一个时间点 = 一个活跃 heavy run
- 浏览不断
- 推理串行
- 可取消
- 可重试
- 可恢复

---

## 17. 下一份文档最适合写什么

在继续只做宏观文档、不开始施工的前提下，下一份最适合写：

1. `interaction-protocol-spec-v1`
   - 各模式输入 / 输出契约
2. `workspace-and-permission-spec-v1`
   - 私有层 / 团队层 / 公共层
3. `precomputed-llm-artifacts-strategy-v1`
   - 是否要提前用 LLM 跑全量材料生成衍生文档

如果只能先写一个，我建议：
- `precomputed-llm-artifacts-strategy-v1`

因为它直接决定后续系统是否要提前生成大量中间文档资产。
