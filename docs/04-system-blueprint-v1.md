# ISMISM 系统蓝图（v1）

## 0. 本文要回答的问题

这份蓝图先不讨论具体 Web 技术栈，而只回答三件事：

1. 这个系统到底有哪些交互形式
2. LLM 在系统里扮演什么角色
3. 整个系统应当按什么结构来搭

一句话总纲：

> ISMISM 应被设计成一个以 LLM 为核心驱动、以“学习—判定—协作—推演”为主循环的交互式理论学徒系统。

它不是静态知识库，也不是普通聊天机器人，而是一个“理论操作环境”。

---

## 1. 系统目标

### 1.1 总目标
让用户能够：
- 学会 ISMISM 的四维框架
- 使用这套框架对对象进行 provisional diagnosis（暂定诊断）
- 看到候选判断、边界差异、证据与反证
- 与 agent / 学者共同修正判断
- 把理论判断继续推到主体、组织与实践层

### 1.2 系统不追求的目标
- 不追求百科式全知回答
- 不追求一键给出唯一正确答案
- 不追求把所有语料平铺成页面
- 不追求单轮“问一句答一句”的弱交互

### 1.3 系统成功标准
成功不看页面数，而看四类能力是否增长：
- 学习能力
- 判定能力
- 协作能力
- 推演能力

---

## 2. 用户与角色模型

系统至少面向四类角色：

### A. 学徒 / 初学者
需求：
- 进入框架
- 学会基本词汇
- 理解 1/2/3/4 与四维判定
- 避免常见误判

### B. 研究者
需求：
- 用系统判定文本、思想、姿态、组织、实践路径
- 获得候选结果与边界分析
- 快速定位证据

### C. 学者 / 作者
需求：
- 与系统来回对话
- 修正、挑战、重排框架
- 积累 canonical case / disputed case
- 推进体系自身发展

### D. Agent
需求：
- 以 ISMISM 规则为工作模式而不是普通聊天模式
- 在 scholar-in-the-loop 条件下做证据优先、候选优先、可修正输出

---

## 3. 核心设计原则

### 原则 1：LLM 是推理核心，不是全部系统
LLM 负责：
- 理解用户意图
- 组织多步对话
- 生成候选判断
- 解释边界差异
- 汇总证据
- 输出结构化报告

但系统还必须提供稳定外壳：
- 语料检索
- 方法规则
- 案例库
- 交互状态机
- 请求锁
- 记录与回放

### 原则 2：先给候选，再给裁决
系统默认输出：
- 第一候选
- 第二候选
- 关键分歧点
- 需要补证据的地方

而不是直接宣布“唯一答案”。

### 原则 3：证据优先于口才
所有高级输出都应能回到：
- 语料片段
- 方法规则
- 边界规则
- canonical case
- confusion pair

### 原则 4：对话是工作流，不是装饰
系统中的交互不是“聊天 UI 包着个回答器”，而是理论判断流程本身。

### 原则 5：一个线程同一时刻只允许一个 LLM 生成任务
在同一会话/线程里，LLM 回复完成之前，禁止重复发请求。
这是硬规则，不是体验建议。

## 4. 系统中的六种主要交互形式

这六种交互形式应作为产品主入口，而不是附属功能。

## 4.1 导学模式（Curriculum Mode）

目标：
- 让用户按正确顺序进入体系

输入：
- 用户水平（初学 / 中级 / 高级）
- 学习目标（框架入门 / 哲学主干 / 符号学转向 / 实践单元）

输出：
- 学习路径
- 节点推荐
- 先修概念
- 高混淆点提醒
- 学后自测

典型流程：
1. 识别用户水平
2. 给出推荐起点
3. 给出 3-7 个学习节点
4. 每学一节点后给一个小测或对比题
5. 决定是否进入下一节点

这个模式不是纯展示内容，而是“带路”。

---

## 4.2 地图模式（Map Mode）

目标：
- 让用户把抽象体系看成可导航结构

主要对象：
- 1/2/3/4 主干
- 四维结构
- 一级/二级/三级节点
- 方法主轴
- 高混淆边界
- 跨区跃迁

用户操作：
- 从宏观图进入细部
- 查看某节点的上下游
- 查看该节点常与哪些节点混淆
- 查看该节点与哪些案例关联

这个模式适合：
- 先建立结构感
- 快速定位语料位置
- 帮助 agent 在对话中说明“你现在站在哪个位置上”

---

## 4.3 判定模式（Diagnosis Mode）

目标：
- 输入一个对象，输出 provisional diagnosis

可判对象：
- 思想家
- 主义
- 文本
- 主体姿态
- 组织形式
- 实践路径
- 现实事件的意识形态结构

输出格式建议固定：
- 对象
- 判定层级（总位 / 四维 / 子位）
- 第一候选
- 第二候选
- 关键分歧点
- 证据片段
- 排除理由
- 不确定性
- 下一步建议

这是系统的主工作模式之一。

---

## 4.4 比较模式（Compare Mode）

目标：
- 处理“像 A 还是像 B”这类问题

适合场景：
- 高混淆格位
- 学者争议
- 用户有先验判断但不确定

输出应强调：
- A 和 B 的共同点
- A 和 B 的真正分野
- 决定分野的关键维度
- 典型误判原因
- 哪类证据最能拉开差距

这个模式可以独立存在，也可以作为判定模式的第二步。

---

## 4.5 案例训练模式（Case Training Mode）

目标：
- 训练用户自己做判断

案例类型：
- canonical case
- confusion pair
- hard case
- disputed case
- adversarial case（故意迷惑人的）

训练流程：
1. 系统给案例
2. 用户先自己判
3. 系统展示候选与理由
4. 用户对比自己哪里偏了
5. 系统生成针对性反馈

这一模式决定系统能不能真正“教会人”。

---

## 4.6 学者协作模式（Scholar Mode）

目标：
- 让 agent 作为学者的协作界面，而不是代替学者给最终裁决

适合场景：
- 框架修正
- 新案例纳入
- 争议位置讨论
- 方法边界澄清
- 新旧版本对照

系统行为：
- 不默认讲基础常识
- 先抓争议点
- 允许 provisional judgment
- 鼓励学者纠偏
- 显式记录“当前结论仍有争议”

这是 ISMISM 与普通聊天产品最不一样的入口。

---

## 5. 一个完整会话的标准流程

无论用户从哪个入口进入，高级交互都应遵循统一流程。

### Step 1：识别任务类型
系统先判断当前是：
- 学习
- 导航
- 判定
- 比较
- 训练
- 学者协作

### Step 2：锁定对象
系统要问清：
- 判的是什么？
- 是判整体还是判某一维？
- 是要快速答案还是要严谨推演？

### Step 3：召回相关材料
系统应从内部材料池中拉取：
- 相关语料段
- 方法规则
- 边界规则
- 类似案例
- 对立案例

### Step 4：生成 provisional output
输出：
- 候选 1
- 候选 2
- 分歧点
- 证据
- 需要进一步确认的问题

### Step 5：等待用户/学者反馈
用户可能：
- 接受
- 质疑
- 要求更细
- 改判定层级
- 引入新证据

### Step 6：二次收敛
系统再更新：
- 候选排序
- 证据权重
- 争议说明
- 下一步建议

## 6. LLM 在系统中的具体角色

LLM 不是一个统一黑箱，而应该在系统里被看成四个子角色。

### 6.1 导师角色
负责：
- 解释概念
- 组织路径
- 提示误区

### 6.2 诊断角色
负责：
- 给出候选格位
- 比较边界
- 解释排除理由

### 6.3 协作角色
负责：
- 接收学者纠偏
- 重排候选
- 识别争议中心

### 6.4 报告角色
负责：
- 把长推理压缩成结构化可读输出

在实现时，前端可以还是一个对话界面，但后端内部要明确这些角色差异。

## 7. 系统内部模块蓝图

建议分成七个层。

## Layer 1：Corpus Layer
内容：
- 总 PDF
- 目录索引
- `split_md/`
- `split_md_clean/`

作用：
- 原始证据与上下文来源

## Layer 2：Method Layer
内容：
- 四维规则
- 1/2/3/4 主干
- 边界规则
- 实践单元
- 术语表

作用：
- 为 LLM 提供稳定方法骨架

## Layer 3：Case Layer
内容：
- canonical cases
- confusion pairs
- hard cases
- disputed cases

作用：
- 支持训练、比较与诊断

## Layer 4：Dialogue Orchestration Layer
内容：
- 会话状态
- 任务识别
- 追问逻辑
- 多轮收敛

作用：
- 让对话成为工作流

## Layer 5：LLM Runtime Layer
内容：
- prompt 编排
- 检索注入
- 结构化输出
- 引用证据

作用：
- 驱动系统推理

## Layer 6：Session State Layer
内容：
- 当前对象
- 当前模式
- 当前候选
- 当前争议点
- 当前活跃生成任务

作用：
- 保证线程一致性和防重复请求

## Layer 7：UI Layer
先不定 Web 细节，但必须支持：
- 模式切换
- 结构导航
- 对话区
- 证据面板
- 候选卡片
- 锁状态显示

## 8. 请求状态机：禁止重复发请求

这是必须落实的硬规则。

## 8.1 线程级规则
同一线程（thread/session）在任一时刻只允许一个活跃 LLM 生成任务。

也就是说：
- 如果当前还在生成/流式输出
- 用户不能在同一线程里再次发请求
- 系统必须直接阻断，而不是默默排队再发第二次请求

## 8.2 状态定义
建议线程状态机如下：

- `idle`
  - 空闲，可发请求
- `preparing`
  - 正在整理上下文、检索、拼 prompt
- `generating`
  - 已经发给 LLM，正在等待首 token
- `streaming`
  - 正在流式返回
- `finalizing`
  - 正在整理最终结构化输出、保存会话状态
- `done`
  - 本轮完成，可发下轮请求
- `error`
  - 本轮出错，可重试
- `cancelled`
  - 用户中止，可重新发起

## 8.3 前端行为规则
当线程状态处于：
- `preparing`
- `generating`
- `streaming`
- `finalizing`

前端必须：
- 禁用发送按钮
- 禁用 Enter 快捷发送
- 输入框可编辑，但不能提交
- 显示“正在回答”状态
- 提供“停止生成”按钮

## 8.4 后端保护规则
不能只靠前端禁按钮，后端也要硬保护。

后端应维护：
- `active_run_id`
- `thread_id`
- `status`
- `started_at`
- `idempotency_key`

若同一 `thread_id` 收到新的推理请求，而状态仍为活跃中，后端返回：
- `409 Conflict` 或等价业务错误
- 附带当前 `active_run_id`
- 告诉前端“已有进行中的生成任务”

## 8.5 重复点击与网络抖动
为防止用户双击、重复点击、网络重传，前端每次发请求都带：
- `idempotency_key`

后端规则：
- 同一 `idempotency_key` 的重复请求，不创建新 LLM 调用
- 只返回现有 run 的状态或流句柄

## 8.6 是否允许排队
当前版本建议：
- 不允许线程内排队

理由：
- 理论判定强依赖上下文
- 排队会让用户误以为第二个问题已经基于第一个答案展开
- 但实际上第一轮还没完成，第二轮的语义是不稳定的

所以最稳妥的是：
- 一个线程，一次一个请求
- 要么等
- 要么取消当前轮
- 再发下一轮

## 9. 非 Web 层面的交互输出组件

即便先不做 Web，也应该先定义产品级输出组件。

### 9.1 候选卡（Candidate Card）
字段：
- 候选名
- 置信度区间（不是绝对概率）
- 支持证据
- 反向证据
- 为何不是另一候选

### 9.2 证据卡（Evidence Card）
字段：
- 来源文件
- 段落摘录
- 对应维度
- 使用原因
- 风险说明（是否可能被误读）

### 9.3 边界卡（Boundary Card）
字段：
- 容易混淆的两个位置
- 真正差别
- 快速判别标准
- 常见误判案例

### 9.4 训练反馈卡（Training Feedback Card）
字段：
- 你判了什么
- 系统候选是什么
- 你错在什么地方
- 下次要先看哪类证据

### 9.5 学者修正卡（Scholar Revision Card）
字段：
- 原判断
- 学者修正意见
- 修正理由
- 是否形成新 canonical case
- 是否形成新边界规则

## 10. 最小可运行系统（MVP）应该怎么切

### MVP-1：学习主循环
有：
- 导学模式
- 地图模式
- 基本术语页
- 最小自测

### MVP-2：判定主循环
有：
- 判定模式
- 固定诊断报告格式
- 证据片段显示
- 候选 1 / 候选 2 输出

### MVP-3：交互主循环
有：
- 线程状态机
- 活跃请求锁
- 停止生成
- 模式切换
- 结构化消息记录

### MVP-4：学者协作主循环
有：
- scholar mode
- 纠偏记录
- 争议点记录
- case 升格机制

只要这四个循环打通，系统就已经成立。

---

## 11. 后续可以再加但不是第一优先级的能力

- 多模态页面（图谱、卡片、结构图）
- 可视化知识地图
- 自动案例生成
- 多模型对照诊断
- 用户画像与学习进度追踪
- 细粒度权限系统
- 团队协作审阅

这些都重要，但都排在“核心交互主循环成立”之后。

---

## 12. 一句话总结蓝图

这个系统最合适的最终形态不是：
- 一个能回答问题的网站

而是：
- 一个以 LLM 为核心推理引擎、以线程级单请求锁为交互纪律、以学习/判定/协作/推演为四大主循环的理论操作环境。

也就是：

> ISMISM = LLM 驱动的交互式理论学徒与诊断工作台。
