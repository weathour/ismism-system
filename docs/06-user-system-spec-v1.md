# ISMISM 用户系统规格（v1）

## 0. 这份文档回答什么问题

这份文档只讨论产品层面的“用户系统”，不讨论具体数据库表设计、认证服务、前端状态管理实现或后端 API 细节。

它回答的问题是：
- 为什么这个系统必须有用户系统
- 用户初始化（onboarding）应该怎么做
- 应该保存哪些用户相关信息
- 哪些信息不应该保存
- 用户状态与理论真相、会话状态、共享知识之间如何分层

一句话结论：

> ISMISM 不能只有“语料 + LLM + 地图”，还必须有一个明确的用户系统，用来承载初始化、持续学习状态、个体偏好、工作记忆与协作边界。

---

## 1. 为什么必须有用户系统

如果没有用户系统，这个产品会立刻退化成：
- 一个只能单次问答的聊天界面
- 一个无法记住学习进度的系统
- 一个无法区分新手、研究者、学者的统一壳子
- 一个无法把地图掌握度、训练记录、学者纠偏关联起来的半成品

而 ISMISM 明显需要：
- 长期学习
- 多轮判定
- 历史纠偏
- 案例积累
- 不同用户档位
- 学者协作

所以用户系统不是附属功能，而是这个产品的基础结构之一。

---

## 2. 用户系统的总目标

用户系统需要同时解决五个问题：

1. 用户是谁
2. 用户想用系统干什么
3. 用户已经学到哪一步
4. 用户当前正在做什么任务
5. 用户与共享体系之间的边界是什么

因此用户系统不是单一“账号系统”，而是五层组合：
- 身份层
- 初始化层
- 学习层
- 工作层
- 权限与隐私层

---

## 3. 用户类型设计

建议至少支持六类用户类型。

## 3.1 游客（Guest）

特点：
- 不登录或弱登录
- 可临时体验导学、地图、部分判定
- 默认不做长期持久化

适用：
- 初次进入系统的人
- 想先试用的人

价值：
- 降低进入门槛

限制：
- 不应默认保存长期学习进度
- 不应开放高价值协作功能

## 3.2 学徒（Learner）

特点：
- 主要使用导学模式、地图模式、案例训练模式
- 需要长期保存掌握进度与训练记录

适用：
- 新手
- 正在系统学习 ISMISM 的用户

## 3.3 研究者（Researcher）

特点：
- 高频使用判定模式、比较模式
- 需要保存判定历史、常用对象、案例工作流

适用：
- 想把 ISMISM 当分析工具使用的人

## 3.4 学者 / 作者（Scholar）

特点：
- 需要纠偏、修正规则、定义 canonical case
- 需要把个人判断升格为共享结构的一部分

适用：
- 作者本人
- 熟悉体系并有修正权限的学者

## 3.5 运营 / 维护者（Operator / Admin）

特点：
- 管理用户、工作区、公开案例、权限策略、导学路径

## 3.6 Agent 账户（System / Agent Identity）

特点：
- 不是人类用户，但在系统中也应有独立身份
- 用于记录哪个 agent 发起了某次诊断、生成了某个案例草稿、提出了哪些候选判断

原因：
- agent 不是匿名后台，它是协作参与者

---

## 4. 初始化（Onboarding）设计

初始化不应过重，否则会把用户吓跑；但也不能太轻，否则系统无法个性化。

建议采用“两段式初始化”。

## 4.1 第一段：极简初始化

第一次进入系统时，只问最少的问题：

- 你的身份更接近哪一类？
  - 游客 / 学徒 / 研究者 / 学者
- 你来这里主要想做什么？
  - 学习框架
  - 进行判定
  - 与系统讨论案例
  - 协作修正体系
- 你对 ISMISM 的熟悉程度？
  - 完全陌生
  - 略知一二
  - 看过部分材料
  - 已熟悉主干
- 你希望系统默认怎样和你说话？
  - 导师式
  - 简洁研究式
  - 学者协作式

这一步的目标不是建立完整画像，而是让系统先知道：
- 用什么入口接待你

## 4.2 第二段：可选初始化 / 校准

在用户愿意继续使用时，再做更细校准：

- 是否做一个最小基线诊断？
- 你最想先掌握哪部分？
  - 1/2/3/4 主干
  - 四维框架
  - 符号学
  - 实践单元
  - 判定方法
- 你想优先分析什么对象？
  - 思想家
  - 文本
  - 主体姿态
  - 组织 / 实践
- 你是否希望系统记住你的学习进度与训练结果？
- 你是否允许系统根据你的历史表现推断掌握程度？

## 4.3 初始化完成后的产物

初始化之后，系统应生成一张“用户启动卡（User Boot Card）”。

启动卡不是公开页面，而是内部产品对象。

它至少应包含：
- 用户类型
- 主要目标
- 初始交互模式
- 初始熟悉度
- 是否已校准
- 是否允许长期持久化

---

## 5. 用户系统中应该保存哪些信息

下面按层来拆。

## 5.1 身份与账户信息（Identity Layer）

建议保存：
- `user_id`
- `display_name`
- `role_type`
- `account_status`
- `created_at`
- `last_active_at`
- `auth_provider`（如果有）

说明：
这是最基础的一层，用来确认“谁在使用系统”。

## 5.2 初始化与目标信息（Onboarding Layer）

建议保存：
- 初始用户类型
- 初始目标
- 初始熟悉度
- 是否完成基线校准
- 当前主学习目标
- 当前主使用目标
- 默认交互档位
- 语言偏好
- 详细程度偏好（简洁 / 解释型 / 学者型）

说明：
这层决定系统一开始怎么接待你。

## 5.3 学习状态信息（Learning State Layer）

建议保存：
- 每个方法节点的掌握等级
- 每个高价值节点的掌握等级
- 掌握等级来源（自评 / 系统推断 / 训练记录）
- 最近复习时间
- 最近训练时间
- 最近卡住的节点
- 高频混淆节点对
- 学习路径完成情况
- 是否完成过某阶段课程

说明：
这是地图“学习状态层”的直接数据来源。

## 5.4 训练与判定历史（Work Memory Layer）

建议保存：
- 已完成的训练案例
- 已做过的判定任务
- 已看过的对照案例
- 历史候选输出摘要
- 历史最终结论摘要
- 用户曾经反复误判的边界
- 系统给过的关键纠偏建议

说明：
这层不只是历史记录，而是后续个性化教学与纠偏的依据。

## 5.5 交互偏好（Interaction Preference Layer）

建议保存：
- 喜欢的交互模式（导学 / 判定 / 比较 / scholar）
- 证据显示偏好（简略 / 默认展开 / 证据优先）
- 是否喜欢系统反问
- 是否接受 provisional judgment
- 是否默认显示对照候选
- 是否默认显示边界卡
- 是否默认显示推演结果

说明：
这层直接影响 LLM 输出风格和界面默认状态。

## 5.6 协作与产出信息（Collaboration Layer）

建议保存：
- 用户创建的书签节点
- 用户收藏的案例
- 用户保存的判定报告
- 用户提交的修正意见
- 用户认可 / 否决的候选判断
- 用户提出的新案例草案
- scholar 审定记录

说明：
这层让系统不仅能学，还能积累用户自己的工作成果。

## 5.7 权限与可见性信息（Permission Layer）

建议保存：
- 用户属于哪些工作区 / 项目空间
- 用户的权限级别
- 哪些内容是私有的
- 哪些内容是团队共享的
- 哪些内容允许升格为系统公共资源

说明：
这个系统未来很可能有“私人学习层”和“共享理论层”并存，因此权限必须前置设计。

## 5.8 隐私与保留策略（Privacy Layer）

建议保存：
- 是否允许保存学习进度
- 是否允许保存训练历史
- 是否允许系统据此推断掌握度
- 是否允许团队共享你的案例
- 是否允许将你的修正意见纳入公共规则草案
- 数据导出偏好
- 数据删除偏好

说明：
用户系统不是为了“多收集数据”，而是为了有边界地支持长期学习与协作。

---

## 6. 哪些信息不应该保存

这是非常重要的一节。

## 6.1 不应保存完整 chain-of-thought

系统可以保存：
- 结构化结论
- 候选结果
- 证据摘要
- 纠偏结论

但不应把内部推理链条原样长期保存为用户可见或系统永久资产。

## 6.2 不应保存所有键盘输入细节

不需要保存：
- 每次编辑中的中间草稿
- 每个字符级输入轨迹
- 用户反复删改的全过程

除非将来专门做写作研究产品，否则没必要。

## 6.3 不应保存重复的大 prompt 上下文副本

不应把每轮完整上下文都长期复制保存。
更合理的是保存：
- 会话摘要
- 关键状态
- 关键判断结果

## 6.4 不应把系统猜测直接当成永久用户事实

例如：
- “系统推断你已熟悉某节点”
不能直接写成不可变事实。
它必须带来源和时间戳。

## 6.5 不应保存与系统价值无关的敏感私密信息

例如：
- 与学习和协作无关的个人隐私
- 无业务必要的身份属性

---

## 7. 用户状态与其他状态的边界

这是系统设计里最容易混的地方。

建议严格区分四类状态。

## 7.1 理论真相状态
例如：
- 节点结构
- 方法规则
- canonical cases
- 边界关系

特点：
- 不属于任何个人
- 不被掌握度污染

## 7.2 用户状态
例如：
- 掌握程度
- 学习路径
- 偏好
- 书签
- 训练记录

特点：
- 是个人化的
- 允许因用户不同而不同

## 7.3 会话状态
例如：
- 当前线程模式
- 当前候选
- 当前正在分析的对象
- 当前活跃 run

特点：
- 是短期状态
- 强依赖当前线程

## 7.4 共享工作区状态
例如：
- 学者团队共建的案例
- 团队认可的修正规则
- 工作区内共享书签和任务板

特点：
- 介于个人与公共之间

这四层必须严格分开，否则产品会混乱。

---

## 8. 用户初始化后的默认体验应该是什么

初始化完成后，系统不应把用户直接扔进一个空聊天框。

建议进入一个“个人工作台首页”，首页至少显示：
- 你的当前身份档位
- 你的当前主目标
- 推荐入口（导学 / 判定 / 案例训练 / scholar）
- 你的学习地图摘要
- 你最近卡住的节点
- 你最近一次未完成任务
- 当前推荐继续点

也就是说，用户系统应当把“空系统”变成“有连续性的工作台”。

---

## 9. 用户系统与地图系统的关系

地图系统依赖用户系统提供个体状态。

具体来说：
- 地图真相层来自理论系统本身
- 地图学习层来自用户系统中的学习状态信息
- 地图任务层来自会话状态系统

因此地图和用户系统是两个模块，但关系非常紧密。

可理解为：
- 地图 = 结构外壳
- 用户系统 = 个体状态供给器

---

## 10. 用户系统与 LLM 的关系

LLM 需要读取用户系统，但要克制读取。

建议读取优先级如下：

### 高优先级可读
- 用户类型
- 当前模式偏好
- 当前学习目标
- 当前掌握状态摘要
- 当前高频混淆点

### 中优先级按需读
- 历史训练记录摘要
- 历史判定报告摘要
- 最近 scholar 修正摘要

### 低优先级默认不读
- 全量历史对话
- 全量个人笔记
- 大量旧任务原文

原因：
LLM 需要“足够知道这个人是谁”，但不应被用户历史淹没。

---

## 11. 多用户协作时的关键问题

如果未来系统允许多人使用，必须明确三件事：

### 11.1 用户的私有层
只对自己可见：
- 自评掌握度
- 私人书签
- 私人训练结果
- 未公开判定草稿

### 11.2 团队共享层
对工作区成员可见：
- 共享案例
- 争议讨论
- 团队修正意见
- 协作任务

### 11.3 公共理论层
对所有用户可见：
- 经过审定的方法节点
- 经过确认的 canonical cases
- 稳定边界规则

如果这三层不分，学者协作会非常混乱。

---

## 12. 用户系统中的最低必要对象

如果只做 MVP，用户系统至少要有这六个对象：

1. `UserProfile`
2. `UserOnboardingState`
3. `UserLearningState`
4. `UserInteractionPreferences`
5. `UserWorkArtifacts`
6. `UserPrivacySettings`

其中最小必做是前四个。

---

## 13. 用户系统当前最重要的产品结论

如果只说一句话：

> 用户系统的本质不是“账号注册”，而是让系统知道“这个人是谁、处在哪一步、正在做什么、哪些东西该记住、哪些东西不该越界”。

这决定了 ISMISM 能不能从一个智能演示品，变成一个真正可长期使用的理论工作台。

---

## 14. 下一份文档最适合写什么

在继续只做宏观文档、不开始施工的前提下，下一份最适合写：

1. `session-and-run-state-spec-v1`
   - 线程锁
   - 活跃 run
   - 重复请求拦截
   - cancel / retry / idempotency

2. `interaction-protocol-spec-v1`
   - 各模式输入输出契约

3. `workspace-and-permission-spec-v1`
   - 私人层 / 团队层 / 公共层如何划分

如果只能先写一个，优先建议：
- `session-and-run-state-spec-v1`

因为它直接决定整个产品的交互纪律。
