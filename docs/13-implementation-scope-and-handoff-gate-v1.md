# ISMISM 实现范围与交施工方闸门规格（v1）

> 更新说明（2026-04-21）：当前前端外显方案已转向“地图 + 聊天窗口”，本文件中的多页面显式工作台方案应视为历史 v1 外显方案；内部系统能力与约束仍保留。详见 `docs/16-frontend-pivot-map-chat-codex-v1.md`。

## 0. 这份文档讨论什么

这份文档只讨论：
- 第一阶段到底应该施工什么
- 哪些东西现在明确不做
- 哪些资产先手工 / 半手工就足够
- 在什么条件下可以把项目正式交给施工方

它不讨论：
- 某个前端框架或后端框架选型
- 具体数据库 schema
- 某个任务队列的实现细节
- 某个部署环境的 DevOps 方案

一句话结论：

> ISMISM 现在可以正式进入“按 v1 范围施工”的阶段，但只能按明确收缩后的 MVP / v1 边界开工，不能把它误做成一个全量自动化、全功能、全语料完成态系统。

---

## 1. 为什么必须写这份范围闸门文档

如果没有这一层，施工方极容易出现三种偏航：

### 偏航 1：先做一个看起来很像产品的聊天壳
表现：
- 首页一个聊天框
- 随便挂个地图入口
- 其它模式都靠 prompt trick 临时拼

结果：
- 线程、地图、证据、关系资产之间没有稳定骨架
- 系统被误建成普通问答产品

### 偏航 2：先做一个“全图谱 / 全语料 / 全摘要”大工程
表现：
- 试图先把 300+ 主题和所有关系一次性结构化
- 追求“先把所有内容自动生成完”

结果：
- 工期被拖爆
- 资产污染严重
- 产品上反而没有一个可稳定使用的入口

### 偏航 3：先堆局部花哨功能
表现：
- 做漂亮图谱动画
- 做复杂多用户协作
- 做推荐算法
- 做移动端

结果：
- 核心训练、判定、审核链条仍未闭合

所以：
范围闸门的作用，不是保守，而是确保施工从真正的系统核心开始。

---

## 2. 当前建议的施工目标

当前最适合交给施工方的，不是“完整终局版 ISMISM”，而是：

## 2.1 v1 目标形态

建设一个可实际运行的 ISMISM 理论工作台 MVP / v1，具备以下最小闭环：
- 用户可以进入统一工作台
- 用户可以浏览地图并打开节点 / 关系 / 证据
- 用户可以在线程中发起导学 / 诊断 / 比较类任务
- 每个线程只允许一个 active heavy run
- 系统可以把结果渲染成结构化对象，而不是纯长文本
- 资产有 draft / reviewed / canonical 等状态
- scholar / review 角色可以处理一批种子资产
- 训练页可以基于一小批案例跑通“做题 -> 反馈 -> 回结构补课”闭环

也就是说：
第一版的目标不是“完成整个 ISMISM”，而是“把 ISMISM 的工作方式跑通”。

## 2.2 当前不要求达成的终局目标

第一版不要求：
- 全量语料都完成稳定结构化
- 所有模式都成熟
- 所有关系资产都预生成完毕
- 所有审核工作流都自动化
- 所有 scholar 协作机制都完备

---

## 3. v1 必须交付的系统能力

下面这些能力，我认为属于“第一版必须有”。

## 3.1 统一工作台壳层

必须有：
- 登录后工作台首页
- 左侧一级导航
- 中央主工作区
- 右侧证据 / 资产 / 洞察面板
- 顶部线程 / run 状态区
- 可见的 run 状态与线程锁提示

原因：
如果没有统一壳层，后面所有模式都会变成拼装页。

## 3.2 地图主入口

必须有：
- 真相层基础地图
- 用户状态层的最小显示能力
- 当前任务层的最小高亮能力
- 点击节点 / 关系后打开详情
- 从地图直接发起导学 / 诊断 / 比较线程

原因：
地图是系统认知入口，不是后加装饰。

## 3.3 通用线程工作区

必须有：
- 线程列表或最近线程入口
- 单线程页
- 结构化输出渲染区
- 当前上下文对象列表
- run 状态
- cancel / retry
- 历史输出最小回看能力

建议让以下任务共享这一骨架：
- orient
- diagnose
- compare
- trace
- tension
- project（至少预留）

## 3.4 结构化输出渲染器

必须能渲染的对象至少包括：
- 节点卡
- Route Card
- Tension Card
- Boundary Card
- 证据锚点
- 候选位 / 排除链 / 不确定性说明

如果第一版只能渲染聊天文本，那么等于还没真正进入 ISMISM 的产品形态。

## 3.5 运行纪律系统

必须有：
- 单线程单 active heavy run 锁
- 显式状态展示
- 用户可理解的失败 / 取消 / 重试状态
- 浏览动作与重推理动作的边界

这条是硬约束，不是优化项。

## 3.6 最小资产生命周期系统

必须有：
- asset status 显示
- draft / reviewed / canonical 基础状态
- 资产详情中能看到来源与证据
- 最小审核动作

说明：
第一版可以不一次做全 `deprecated / superseded / rejected` 流程自动化，但状态模型必须预留且基本可表达。

## 3.7 最小训练闭环

必须有：
- 一批种子训练案例
- 用户提交判断
- 系统给出反馈与纠偏
- 反馈可指向地图或导学路径

如果没有训练闭环，系统会退化成“会解释的结构浏览器”。

## 3.8 最小 scholar / review 工作台

必须有：
- 一批待审核资产列表
- 打开资产详情
- 查看证据锚点
- 标记 reviewed / canonical / rejected（至少前两者必须稳）
- 留 review note

原因：
如果第一版没有审核面，前面的资产生命周期就只是纸面规则。

## 4. v1 推荐采用的“种子驱动”策略

第一版不要等全量语料完工。

建议采用“种子驱动”方式：
- 先选一小批高价值节点
- 先选一小批高价值关系型资产
- 先选一小批训练案例
- 先让这些对象在地图、线程、训练、审核四个面里跑通

### 4.1 推荐种子范围

建议第一批只做：
- 20–40 个核心节点
- 30–80 个高价值关系型资产
- 10–20 个 canonical / confusion / hard / disputed case
- 一小批高质量证据锚点

数量不是硬指标，关键是：
这些对象必须足够支撑真实使用，而不是只够做 demo 截图。

### 4.2 第一批对象的选择原则

优先选：
- 方法骨架核心节点
- 高混淆边界
- 可体现运动 / 矛盾 / 中介的典型关系
- 能代表实践单元方向的对象
- 适合训练纠偏的案例

不优先选：
- 大量边缘节点
- 只有摘要价值、没有关系价值的对象
- 需要大规模自动化后才有意义的花哨功能

## 5. v1 明确不做的事情

下面这些我建议明确列入第一版非目标（out of scope）。

## 5.1 不做“全量自动资产化”

不要求：
- 全量节点卡自动生成
- 全量关系卡自动生成
- 全量训练题自动生成
- 全量 scholar 初审自动化

原因：
会把施工重点带偏到大规模流水线，而不是产品闭环。

## 5.2 不做复杂多租户 / 多组织协作

第一版可以先假定：
- 单一系统
- 少量高权限维护者
- 少量 scholar / researcher 角色

不需要一开始就做成完整平台级协作系统。

## 5.3 不做移动端优先

第一版优先桌面工作台体验。

原因：
地图、证据、关系卡、审核面天然更适合大屏。

## 5.4 不做炫技型图谱编辑器

第一版的地图重点是：
- 稳定浏览
- 稳定高亮
- 稳定跳转
- 稳定打开对象

不必先做：
- 超复杂拖拽建图
- 自由编辑图谱布局
- 多人实时共同编辑图谱

## 5.5 不做深度个性化推荐系统

第一版只需要：
- 最近继续
- 基于当前任务的下一步建议
- 基于错误的补结构建议

不需要复杂推荐引擎。

## 5.6 不把 Atlas 当成第一版主真相层

Atlas 可作为：
- 候选对象来源
- 历史证据层
- 辅助审核材料

但不应强迫第一版先把 Atlas 全面产品化。

## 5.7 不要求第一版解决“最终判定权威性”问题

第一版应实现：
- provisional
- reviewed
- canonical 的流程能力

但不要求第一版就完成所有高争议对象的终极定论。

---

## 6. 第一版建议的数据与资产来源策略

## 6.1 真相层来源

第一版优先使用：
- 已清洗文本 `split_md_clean/`
- 已确认的方法骨架文档
- 手工 / 半手工筛出的节点与关系资产

## 6.2 资产生成方式

建议混合：
- 手工整理
- LLM 草稿生成
- 人工筛选与审核

而不是：
- 先全自动生成
- 再想办法救火

## 6.3 训练案例来源

建议第一批案例：
- 人工精选
- 明确证据锚点
- 可解释错因
- 可回跳到地图 / 导学

## 6.4 scholar / review 数据来源

第一版完全可以接受：
- 审核队列由手工维护或半自动生成
- 一部分状态变更由后台工具辅助
- 不要求一开始就把全部审核流程全自动编排

---

## 7. 推荐的施工顺序

为了防止“每一块都做一点，最后没有任何一块能用”，建议按下面的顺序开工。

## 7.1 Step 1：统一工作台壳层 + 线程 / run 纪律

先完成：
- 登录后工作台骨架
- 顶栏 / 左栏 / 右栏 / 中心区 / run 状态区
- 单线程单 active heavy run 锁
- 基本线程页

这是整个系统的地基。

## 7.2 Step 2：地图主入口 + 对象详情侧板

先完成：
- 地图浏览
- 节点 / 关系点击
- 对象详情面板
- 地图到线程的发起动作

## 7.3 Step 3：结构化输出渲染器 + 导学 / 诊断 / 比较线程模板

让系统第一次真正长成 ISMISM，而不是聊天机器人。

## 7.4 Step 4：最小资产生命周期与 scholar / review 工作台

把：
- draft / reviewed / canonical
- 资产详情
- review note
- 审核动作
跑通。

## 7.5 Step 5：最小训练闭环

把：
- 训练题
- 用户作答
- 纠偏反馈
- 回结构补课
跑通。

## 7.6 Step 6：再扩展更多资产类型与更多模式

例如：
- project
- tension 深化
- scholar 协作增强
- 更多生命周期自动化

## 9. 正式交施工方的闸门条件

我建议用三段式状态，而不是“能不能开工”二分法。

## 9.1 状态 A：探索期（not ready for formal handoff）

特征：
- 只有零散想法
- 页面边界不清
- 线程 / run 纪律不清
- 第一版范围不清
- 种子资产来源未定

这时可以做概念验证，但不应正式全面施工。

## 9.2 状态 B：施工前冻结准备期（freeze-prep）

特征：
- 核心产品骨架已写出
- 但前端信息架构或实现范围仍未冻结
- 仍可能随时大改

这是写完 11 但还没写 12 / 13 时的状态。

## 9.3 状态 C：可正式 handoff 的 v1 状态（ready for scoped build）

需要同时满足：

### 条件 1：核心产品文档已齐备
至少包括：
- 系统蓝图
- 地图规格
- 用户规格
- 会话 / run 规格
- 交互协议规格
- 关系型资产规格
- 生命周期与审核规格
- 前端信息架构规格
- 实现范围与 handoff gate 规格

### 条件 2：第一版页面集合已冻结
必须明确：
- 第一版有哪些页面
- 哪些模式共用线程工作区
- 哪些页面不在第一版

### 条件 3：第一版必须 / 不做清单已冻结
必须明确：
- 哪些能力是硬要求
- 哪些能力是后置增强
- 哪些东西不要提前做

### 条件 4：第一批种子资产策略已明确
至少明确：
- 做哪些节点
- 做哪些关系资产
- 做哪些训练案例
- 它们从哪里来

### 条件 5：运行纪律与生命周期已可转译为产品实现
也就是施工方拿到文档后，不会误以为：
- 全站一次只能有一个 run
- 地图只是附属页面
- 生命周期只是后台备注

## 10. 我对当前状态的判断

在补齐 `docs/12-front-end-information-architecture-v1.md` 与本文件之后，我的判断是：

## 10.1 现在已经可以正式交施工方

但这里的“可以正式交”有明确边界：
- 是可以交付一个 scoped v1 / MVP 的施工任务
- 不是说系统定义已经终局完备
- 不是说后续不会继续补文档或修订资产策略

也就是说：
现在已经达到“可正式启动第一版施工”的条件。

## 10.2 施工方式必须是收缩式开工

正确做法：
- 先按 v1 范围做稳最小闭环
- 先让地图、线程、证据、关系资产、训练、审核跑通
- 再谈扩展

错误做法：
- 把所有未来设想都同时纳入第一版
- 把语料全量完成当成前置条件
- 把所有 scholar 复杂协作一次到位

---

## 11. 交施工方时最该说清楚的一句话

如果只能对施工方说一句总原则，我会说：

> 请先把 ISMISM 做成一个有地图入口、有线程纪律、有结构化输出、有关系型资产、有训练闭环和审核能力的理论工作台；不要先把它做成一个华丽的聊天产品，也不要先把它做成一个试图一次吃下全语料的自动化大工程。

---

## 12. 当前推荐的产品结论

一句话总结：

> 现在的 ISMISM 已经具备正式交施工方的产品文档条件，但 handoff 的对象应当是一个明确收缩后的 v1 / MVP：以统一工作台、地图、线程、结构化输出、运行纪律、最小训练闭环和最小审核链条为核心，而不是终局版全功能系统。
