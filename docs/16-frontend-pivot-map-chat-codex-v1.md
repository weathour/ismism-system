# ISMISM 前端方案转向说明（map + chat + codex in docker）

## 0. 这份文档的作用

这份文档用于记录当前已经确认生效的方案转向：

- **前端显式只保留：大地图 + 聊天窗口**
- 其余 compare / evidence / training / review / 生命周期 / 关系资产等内容，转入**系统内部能力层**
- 前端不再把这些能力做成显式页面族，而是由 LLM 在对话中按需调用和体现

这不是在否定之前的系统文档，而是在明确：

> 当前生效的是一种更收缩的前端表达方式；系统内部能力不取消，只是不再全部前端显式化。

---

## 1. 当前生效的产品理解

ISMISM 仍然不是普通聊天产品。

它仍然是一个：
- 理论学习系统
- 诊断系统
- 学者协作系统

但在当前前端实现中，用户显式面对的只剩两个主界面：

1. **大地图**
2. **聊天窗口**

也就是说，产品外显形态从：
- 工作台首页 / 地图 / 线程 / 训练 / review 多页面

收缩为：
- 地图主入口
- 对话主窗口

而内部系统能力继续存在。

---

## 2. 什么没有变

以下内容**没有被取消**，只是从“前端显式模块”变成“内部查询/调用能力”：

- 四维框架（场域论 / 本体论 / 认识论 / 目的论）
- 节点 / 路线 / 张力 / 中介 / 边界等关系资产
- 证据锚点
- 训练案例
- 审核状态（draft / reviewed / canonical 等）
- 线程级 run 纪律
- 结构化输出协议

所以新的理解不是：
- 这些能力不做了

而是：
- 这些能力**进入内部系统层**，由 codex 在对话中按需查询和调用。

---

## 3. 新前端架构

### 3.1 前端显式层

前端只保留：

#### A. 大地图
职责：
- 结构定位
- 上下文选择
- 节点高亮
- 比较候选选择
- 约束对话的推理范围

#### B. 聊天窗口
职责：
- 学习引导
- 诊断
- 比较
- 证据调用
- 追问与纠偏
- 推荐下一跳

### 3.2 内部系统层

内部保留：
- nodes
- relation assets
- evidence anchors
- training cases
- review states
- structured outputs
- run/thread state

这些对象不一定以显式前端页面出现，但必须能被对话系统查询。

### 3.3 推理/执行层

推理层采用：
- docker 中的 codex

前端通过一个 chat orchestration 边界把：
- 当前地图上下文
- 当前节点
- 当前 compare 对象
- 当前线程状态

传入 codex，由 codex 再去调用内部种子/资产层。

---

## 4. 当前仍必须保留的硬约束

虽然前端收缩了，但以下硬约束仍然成立：

### 4.1 地图不能变成装饰
地图仍然必须是主入口之一，而不是聊天页旁边的附属按钮。

### 4.2 线程级 single active heavy run 仍是硬约束
同一线程同一时刻只允许一个活跃 heavy run。

### 4.3 轻动作仍然要与 heavy action 分开
地图浏览、看节点、看证据预览这类轻动作不应被线程 heavy run 锁死。

### 4.4 内部输出仍要结构化
即使前端最终显示为对话，内部仍必须保留结构化对象：
- candidate outputs
- relation findings
- evidence pack
- uncertainty
- recommended jumps

### 4.5 compare / evidence / training / review 仍然存在
只是它们不一定对应显式前端页面，而是由对话系统调出。

---

## 5. 这样改的收益与风险

### 收益
- 前端更收缩
- 用户入口更统一
- 地图 + 对话成为更稳定的两大外显界面

### 风险
- 非常容易退化成普通聊天壳
- 如果地图约束不够强，内部结构会在对话里被冲淡
- 训练 / 审核 / 生命周期若完全不可见，后期容易失控

因此当前必须坚持：
- 地图强约束
- 内部结构化协议
- 关系资产可查询
- 线程纪律不松动

当前仓库里已经实现/搭出的东西包括：

- 手工 seed 包
- 1/2/3/4 主干节点与二级核心节点
- 关系资产、证据锚点、训练案例、审核状态的内部种子层
- 本地线程模型与线程级 run 锁
- 地图多视图原型
- 四维镜与四个重点专题深描卡

这些内容将继续保留，但在下一阶段会逐步重构成：
- 更收缩的地图 + chat 前端
- 更明确的 codex-in-docker 调用边界

---

## 7. 下一阶段施工重点

接下来优先做：

1. 把前端进一步收缩成地图 + chat 两大主区
2. 把 compare / evidence / training / review 改造成内部 query surface
3. 定义前端 -> orchestration -> codex -> internal seed layer 的调用边界
4. 继续增强地图本身，让它真正承担上下文选择器功能，而不是树状目录

---

## 8. 一句话结论

> 当前生效方案不是“取消系统能力”，而是把它们从显式工作台页面收缩为地图 + 对话所调用的内部能力层；前端只保留大地图和聊天窗口，但系统内部仍必须维持结构化对象、关系资产、证据锚点与线程纪律。
